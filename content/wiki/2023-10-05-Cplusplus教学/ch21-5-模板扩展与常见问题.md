---
title: "模板扩展与常见问题"
---

本节整理使用这个 CMake 模板时最常见的扩展方式和问题排查。

## 添加新的源文件

当前库目录使用：

```cmake
file(GLOB_RECURSE ${PREFIX}_SRC_LIST CONFIGURE_DEPENDS
  "${CMAKE_CURRENT_LIST_DIR}/src/*.c"
  "${CMAKE_CURRENT_LIST_DIR}/src/*.cpp"
)
```

所以只要把新的 `.cpp` 放到对应模块的 `src/` 目录下，CMake 会自动收集。

例如：

```text
src/lib1/src/math_utils.cpp
```

然后重新构建：

```bash
cmake --build --preset linux-debug
```

如果发现新文件没有参与编译，可以手动重新 configure：

```bash
cmake --preset linux-debug
cmake --build --preset linux-debug
```

## 添加新的头文件

推荐路径：

```text
src/lib1/inc/lib1/math_utils.hpp
```

代码中包含：

```cpp
#include "lib1/math_utils.hpp"
```

不要直接放成：

```text
src/lib1/inc/math_utils.hpp
```

因为多个库可能出现同名头文件。使用 `inc/lib1/` 这种结构可以避免冲突。

## 添加新的库模块

假设要添加 `lib3`。

目录结构：

```text
src/lib3/
├── CMakeLists.txt
├── inc/
│   └── lib3/
│       └── example.hpp
└── src/
    └── example.cpp
```

`src/lib3/CMakeLists.txt`：

```cmake
set(PREFIX "lib3")

file(GLOB_RECURSE ${PREFIX}_SRC_LIST CONFIGURE_DEPENDS
  "${CMAKE_CURRENT_LIST_DIR}/src/*.c"
  "${CMAKE_CURRENT_LIST_DIR}/src/*.cpp"
)

add_library(${PREFIX}_src_lib SHARED
  ${${PREFIX}_SRC_LIST}
)

target_include_directories(${PREFIX}_src_lib
  PUBLIC
    $<BUILD_INTERFACE:${CMAKE_CURRENT_LIST_DIR}/inc>
    $<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}>
)

target_link_libraries(${PREFIX}_src_lib
  PUBLIC
    project_options
  PRIVATE
    project_warnings
)

# ========================
# Third-party dependencies
# ========================

# =======
# Install
# =======
install(TARGETS ${PREFIX}_src_lib
  LIBRARY DESTINATION ${CMAKE_INSTALL_LIBDIR}
  ARCHIVE DESTINATION ${CMAKE_INSTALL_LIBDIR}
  RUNTIME DESTINATION ${CMAKE_INSTALL_BINDIR}
)

install(DIRECTORY "${CMAKE_CURRENT_LIST_DIR}/inc/"
  DESTINATION ${CMAKE_INSTALL_INCLUDEDIR}
)
```

然后在 `src/CMakeLists.txt` 中添加：

```cmake
add_subdirectory(lib3)
```

并链接到主程序：

```cmake
target_link_libraries(${PROJECT_NAME}
  PRIVATE
    lib1_src_lib
    lib2_src_lib
    lib3_src_lib
)
```

## 添加新的可执行文件

如果一个项目有多个程序，例如：

```text
src/main.cpp
src/tools/calibrate_camera.cpp
```

可以在 `src/CMakeLists.txt` 里添加：

```cmake
add_executable(calibrate_camera
  ${CMAKE_CURRENT_SOURCE_DIR}/tools/calibrate_camera.cpp
)

target_link_libraries(calibrate_camera
  PRIVATE
    project_options
    project_warnings
    lib1_src_lib
    lib2_src_lib
)

set_target_properties(calibrate_camera PROPERTIES
  INSTALL_RPATH "$ORIGIN/../${CMAKE_INSTALL_LIBDIR}"
)

install(TARGETS calibrate_camera
  RUNTIME DESTINATION ${CMAKE_INSTALL_BINDIR}
)
```

安装后：

```text
install/linux-debug/bin/calibrate_camera
```

## 改项目名

顶层：

```cmake
project(cmake_template VERSION 1.0.0 LANGUAGES C CXX)
```

改成：

```cmake
project(robot_app VERSION 1.0.0 LANGUAGES C CXX)
```

因为主程序使用：

```cmake
add_executable(${PROJECT_NAME}
  ${CMAKE_CURRENT_SOURCE_DIR}/main.cpp
)
```

所以可执行文件会从：

```text
cmake_template
```

变成：

```text
robot_app
```

README 里的运行命令也要同步改：

```bash
./install/linux-debug/bin/robot_app
```

## 改 C++ 标准

当前：

```cmake
set(CMAKE_CXX_STANDARD 17)
target_compile_features(project_options INTERFACE cxx_std_17)
```

如果要改 C++20：

```cmake
set(CMAKE_CXX_STANDARD 20)
target_compile_features(project_options INTERFACE cxx_std_20)
```

如果要改 C++23：

```cmake
set(CMAKE_CXX_STANDARD 23)
target_compile_features(project_options INTERFACE cxx_std_23)
```

同时确认编译器支持对应标准。

## 改动态库为静态库

当前：

```cmake
add_library(${PREFIX}_src_lib SHARED
  ${${PREFIX}_SRC_LIST}
)
```

改成静态库：

```cmake
add_library(${PREFIX}_src_lib STATIC
  ${${PREFIX}_SRC_LIST}
)
```

动态库和静态库对比：

| 类型 | 优点 | 缺点 |
|:---|:---|:---|
| `SHARED` | 可执行文件较小，库可独立更新 | 运行时要能找到 `.so` |
| `STATIC` | 部署简单，很多代码打进可执行文件 | 可执行文件更大，更新库要重新链接 |

模板默认 `SHARED`，是为了演示安装动态库和 RPATH。

## 用 `BUILD_SHARED_LIBS` 控制库类型

也可以不写 `SHARED`：

```cmake
add_library(${PREFIX}_src_lib
  ${${PREFIX}_SRC_LIST}
)
```

然后在 preset 中控制：

```json
"BUILD_SHARED_LIBS": "ON"
```

或：

```json
"BUILD_SHARED_LIBS": "OFF"
```

这样同一个模板可以通过配置切换动态库或静态库。

## 添加宏定义

给某个 target 添加宏：

```cmake
target_compile_definitions(${PREFIX}_src_lib
  PRIVATE
    LIB1_ENABLE_LOG
)
```

C++ 中使用：

```cpp
#ifdef LIB1_ENABLE_LOG
// log code
#endif
```

带值宏：

```cmake
target_compile_definitions(${PREFIX}_src_lib
  PRIVATE
    LIB1_VERSION="1.0.0"
)
```

可见性选择：

| 关键字 | 场景 |
|:---|:---|
| `PRIVATE` | 只影响本库源码 |
| `PUBLIC` | 本库源码和使用者都需要这个宏 |
| `INTERFACE` | 本 target 自己不用，只传给使用者 |

## 添加编译选项

给某个库单独加选项：

```cmake
target_compile_options(${PREFIX}_src_lib
  PRIVATE
    -Wshadow
)
```

不建议用全局：

```cmake
add_compile_options(-Wshadow)
```

因为全局选项会影响所有 target，不利于排查。

## 添加 include 路径

推荐：

```cmake
target_include_directories(${PREFIX}_src_lib
  PRIVATE
    ${CMAKE_CURRENT_LIST_DIR}/some_private_include
)
```

不推荐：

```cmake
include_directories(some_private_include)
```

原因：

1. `include_directories` 是目录级影响，范围更大。
2. `target_include_directories` 能明确说明哪个 target 需要这个路径。

## 添加测试目录

如果以后要加测试，可以新增：

```text
test/
└── CMakeLists.txt
```

顶层添加：

```cmake
include(CTest)

if(BUILD_TESTING)
  add_subdirectory(test)
endif()
```

preset 中可以控制：

```json
"BUILD_TESTING": "ON"
```

不过当前模板要求不新增 examples，测试目录是否添加要看项目需求。

## 清理构建和安装目录

因为模板已经有 `.gitignore` 忽略：

```gitignore
build/
install/
log/
```

所以这些目录都是生成物。

清理 Debug：

```bash
rm -rf build/linux-debug install/linux-debug
```

清理 Release：

```bash
rm -rf build/linux-release install/linux-release
```

重新构建：

```bash
cmake --preset linux-debug
cmake --build --preset linux-debug
cmake --install build/linux-debug
```

## 为什么安装目录里可能是 `lib64`

本模板使用：

```cmake
include(GNUInstallDirs)
```

库安装时使用：

```cmake
LIBRARY DESTINATION ${CMAKE_INSTALL_LIBDIR}
```

在某些 Linux 发行版上：

```text
CMAKE_INSTALL_LIBDIR = lib64
```

所以安装结果是：

```text
install/linux-debug/lib64/
```

这不是错误，而是系统惯例。

不要手写：

```cmake
LIBRARY DESTINATION lib
```

否则会绕开 CMake 对系统目录的判断。

## 运行时报找不到 `.so`

错误类似：

```text
error while loading shared libraries: liblib1_src_lib.so: cannot open shared object file
```

常见原因：

1. 没有执行 `cmake --install ...`。
2. 可执行文件不是从安装目录运行的。
3. `INSTALL_RPATH` 没设置对。
4. 手动移动了 `bin/` 或 `lib64/` 的相对位置。

本模板设置：

```cmake
set_target_properties(${PROJECT_NAME} PROPERTIES
  INSTALL_RPATH "$ORIGIN/../${CMAKE_INSTALL_LIBDIR}"
)
```

所以保持这种结构即可：

```text
install/linux-debug/
├── bin/
│   └── cmake_template
└── lib64/
    ├── liblib1_src_lib.so
    └── liblib2_src_lib.so
```

## CMake 找不到 Eigen 或 OpenCV

先确认安装开发包。

Eigen：

```bash
sudo apt install libeigen3-dev
sudo dnf install eigen3-devel
```

OpenCV：

```bash
sudo apt install libopencv-dev
sudo dnf install opencv-devel
```

再重新 configure：

```bash
cmake --preset linux-debug
```

如果库安装在非标准路径，添加：

```bash
cmake --preset linux-debug -DCMAKE_PREFIX_PATH=/opt/my_library
```

或者写入 preset：

```json
"CMAKE_PREFIX_PATH": "/opt/my_library"
```

## ccache 导致 configure 失败

有些环境中，`gcc`、`g++` 可能指向 ccache 包装器。如果 ccache 目录不可写，CMake 检查编译器时可能失败。

临时解决：

```bash
CC=/usr/bin/gcc CXX=/usr/bin/g++ cmake --fresh --preset linux-debug
```

这只是验证或特殊环境下的处理，不建议写死到模板中。

如果你确实想在 preset 里指定编译器，可以加：

```json
"cacheVariables": {
  "CMAKE_C_COMPILER": "/usr/bin/gcc",
  "CMAKE_CXX_COMPILER": "/usr/bin/g++"
}
```

但这会降低模板通用性，所以默认不写。

## VSCode 里没有识别 preset

检查：

1. 是否安装 CMake Tools 扩展。
2. 打开的目录是否是项目根目录。
3. `CMakePresets.json` 是否在项目根目录。
4. JSON 是否合法。

命令行检查：

```bash
cmake --list-presets
```

如果命令行能看到：

```text
linux-debug
linux-release
```

说明 preset 文件本身没问题。

## VSCode 按 F5 不能调试

F5 调试需要三样东西同时正常：

1. CMake Tools 能识别 preset。
2. Microsoft C/C++ 扩展能启动 GDB。
3. `.vscode/launch.json` 能拿到当前 CMake target 路径。

### 检查 gdb

命令行检查：

```bash
gdb --version
```

没有安装就执行：

```bash
sudo apt install gdb
```

或：

```bash
sudo dnf install gdb
```

### 检查 launch target

模板里的 `launch.json` 使用：

```json
"program": "${command:cmake.launchTargetPath}"
```

这要求 CMake Tools 已经选择了一个可执行 target。

如果 F5 时提示找不到程序，先在 VSCode 里选择：

```text
cmake_template
```

作为 launch target。

### 检查是否已经构建

F5 调试的是 build 目录里的可执行文件，例如：

```text
build/linux-debug/src/cmake_template
```

如果还没有 build，这个文件不存在，F5 就无法启动。

先执行：

```bash
cmake --preset linux-debug
cmake --build --preset linux-debug
```

或者在 VSCode CMake Tools 中执行 Configure 和 Build。

### F5 不需要每次 install

日常开发调试时，F5 通常运行的是 build 目录产物：

```bash
./build/linux-debug/src/cmake_template
```

不是安装目录产物：

```bash
./install/linux-debug/bin/cmake_template
```

`install` 主要用于验证安装布局、头文件安装、动态库 RPATH 等是否正确。

## `.gitignore` 怎样保留 launch.json

模板可以忽略大多数 VSCode 本地配置，但保留 F5 调试入口。

推荐写法：

```gitignore
.vscode/*
!.vscode/launch.json
```

这样：

1. `.vscode/settings.json` 不会进入模板。
2. `.vscode/tasks.json` 不会进入模板。
3. `.vscode/launch.json` 会保留，方便开箱 F5 调试。

## 模板的核心规则

最后总结一下这个模板最重要的几条规则：

1. 顶层 `CMakeLists.txt` 只做总控。
2. 公共编译规则放在 `cmake/ProjectOptions.cmake`。
3. 主程序由 `src/CMakeLists.txt` 管理。
4. 每个库目录自己管理自己的源码、头文件、安装和第三方依赖。
5. 头文件路径带模块名，例如 `lib1/eigen3_test.hpp`。
6. 第三方库放在使用者自己的 `Third-party dependencies` 区块。
7. 构建参数放在 `CMakePresets.json`，不要写死在顶层 CMake。
8. 生成目录 `build/`、`install/`、`log/` 不进入 Git。
9. F5 调试交给 `.vscode/launch.json`，构建仍然交给 CMakePresets。
